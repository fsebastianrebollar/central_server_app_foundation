# 10 · Accesibilidad y motion

No perseguimos conformidad WCAG AAA — buscamos que la app sea **usable por teclado**, **legible para daltónicos** y **respetuosa con `prefers-reduced-motion`**. Lo estrictamente práctico.

## 10.1 · Contraste

### Tokens revisados contra AA

| Par | Ratio | ¿Pasa AA body? | ¿Pasa AA UI? |
|---|---|---|---|
| `--text` sobre `--bg` (claro) | ~12:1 | ✓ | ✓ |
| `--text` sobre `--bg` (oscuro) | ~14:1 | ✓ | ✓ |
| `--text-secondary` sobre `--bg` (claro) | ~4.5:1 | ✓ límite | ✓ |
| `--text-secondary` sobre `--bg` (oscuro) | ~6:1 | ✓ | ✓ |
| `--neon-text` (`#c85500`) sobre `--bg-surface` (claro) | ~4.6:1 | ✓ límite | ✓ |
| `--neon` (`#ff6a00`) sobre `--bg-surface` (oscuro) | ~4.8:1 | ✓ | ✓ |
| `#fff` sobre `--neon` (`#ff6a00`) | ~3:1 | ✗ para texto chico | ✓ para UI grande |
| `#000` sobre `--neon` | ~5.5:1 | ✓ | ✓ |

**Implicaciones**:
- `--neon-text` es el motivo por el que en claro el "color acento como texto" se oscurece a `#c85500`. Si usas `--neon` (`#ff6a00`) como color de texto sobre fondo claro, fallas AA.
- Texto blanco sobre fondo naranja en botones primarios está en el límite. Por eso los botones primarios usan `font-weight: 600` o `700` (texto grande equivalente) para cumplir AA de UI (3:1). **No reduzcas el peso** a 400-500 en botones primarios.
- El logo del header (28×28, texto "C" o similar) usa `#000` sobre naranja: el máximo contraste. Es por eso que es el **único** sitio con texto negro sobre naranja (ver `05-colors.md`).

## 10.2 · Navegación por teclado

### Focus visible en controles custom

Regla: **si haces `outline: none` en un `:focus`, obligatorio poner algo en `:focus-visible`**.

```css
input[type="checkbox"]:focus-visible,
input[type="radio"]:focus-visible {
    outline: 2px solid rgba(255, 106, 0, 0.4);
    outline-offset: 2px;
}

.ws-card:focus-visible,
.report-card:focus-visible {
    outline: 2px solid var(--neon);
    outline-offset: 2px;
}
```

`:focus-visible` sólo se dispara con teclado (no con mouse click) gracias al UA heuristic, por lo que no "contamina" el look al click.

### Botones e inputs estándar

Su `:focus` no se hace `outline: none` — dejamos que el border cambie a `--neon` y añadimos un `box-shadow 3px --neon-subtle` que actúa como "ring". Esto es accesible por teclado sin esfuerzo extra.

### Tab order

- El orden natural del DOM respeta la jerarquía visual.
- No usamos `tabindex > 0` (romper el orden natural causa confusión).
- Cards clicables (workspace cards, report cards) llevan `tabindex="0"` para entrar al flow de tab. Su `:focus-visible` las resalta.
- Modales **deberían** trap el focus — hoy no lo hacemos sistemáticamente. Es deuda conocida; si un modal se vuelve crítico para un usuario teclado-only, añádele `focus-trap` JS.

### Atajos de teclado

- `Esc` cierra modales, popovers, dropdowns (implementado en los handlers JS).
- `Enter` en un input de búsqueda o filter ejecuta el submit.
- No hay shortcuts globales tipo `?`, `/`, `k` — evitado deliberadamente (fábrica, usuarios no técnicos).

## 10.3 · `prefers-reduced-motion`

### Cuándo respetarlo

Las animaciones **ambientales** (las que se reproducen solas o en bucle largo) deben respetarlo. Ejemplo:

```css
@media (prefers-reduced-motion: reduce) {
    .report-card,
    .report-card-glow,
    .report-card-spark-line,
    .report-card-bars span {
        animation: none;
    }
}
```

### Cuándo **no** es necesario respetarlo

Las animaciones de **feedback de estado** (pulso tras click, flash toast entrando) son tan cortas (< 0.3s) que no causan motion sickness. El sistema operativo no las cuenta como problemáticas. **No** las desactivamos salvo que el usuario reporte problemas.

### Regla práctica

- `@keyframes` que dura **< 0.5s y no se repite** → no hace falta gate.
- `@keyframes` con `animation-iteration-count: infinite` o con drift ambiental → **obligatorio** respetar `prefers-reduced-motion`.

## 10.4 · Color y estado

**No comunicar estado solo con color.** Usamos un indicador adicional siempre:

- Fila OK → borde izquierdo verde 5px **y** badge `.badge-ok` con texto.
- Fila FAIL → borde izquierdo rojo 5px **y** badge `.badge-fail`.
- Workspace dirty → punto naranja **y** tooltip "unsaved changes".
- Chart semántico OK/FAIL → usa posición (columnas) + color, no solo color.

Esto cubre daltónicos rojo/verde (protanopia/deuteranopia) que son el caso más común. Si en el futuro añades un chart nuevo, asegúrate de usar **forma o patrón** adicional, no solo color.

## 10.5 · `aria-*` y semántica HTML

Enfoque pragmático: usamos **HTML nativo semántico** donde podemos (`<button>`, `<nav>`, `<main>`, `<aside>`, `<table>`, `<th scope="col">`). Solo añadimos `aria-*` donde la semántica HTML no alcanza:

- Pills-toggle del filter bar: `role="button"` + `aria-pressed="true|false"`.
- Toolbar flotante de vista: `role="tablist"` con `aria-selected`.
- Dropdowns custom (ws-dropdown, lang-dropdown): `aria-expanded` en el trigger, `aria-labelledby` en el panel.
- Toasts: `role="status"` o `role="alert"` según severidad.
- Modales: `role="dialog"` + `aria-labelledby` apuntando al título.

**No abuses de `aria-label`**: si el elemento ya tiene texto visible, es redundante y puede shadowearlo.

## 10.6 · Tamaños mínimos táctiles

En mobile, los touch targets interactivos deben ser **≥ 36×36px**. Actualmente:

| Control | Tamaño | ¿Pasa? |
|---|---|---|
| `.btn-lang`, `.theme-toggle` | 34×34 | Borderline — OK por margen alrededor |
| `.app-header-hamburger` | 36×36 | ✓ |
| `.sidebar-link` | 40+ alto | ✓ |
| `.view-toolbar-btn` | 32 alto | ✗ — compensado por ancho |
| `.fbar-pill` | 26 alto | ✗ en mobile densos |

Para `.fbar-pill` aceptamos la densidad a cambio del alto uso: los usuarios crean muchas condiciones y acomodar 44px por pill rompe la UX. Compensamos con:
- Suficiente gap horizontal entre pills para que los toques no colisionen.
- Un área de tap ligeramente expandida via `padding` (no solo height).

## 10.7 · Screen readers

No optimizamos intensivamente, pero evitamos los antipatrones:

- Iconos decorativos (`//`, flechas de ordenación, el `×` de cerrar) llevan `aria-hidden="true"` cuando el texto contextual ya describe la acción.
- Botones icon-only llevan `aria-label` (p. ej. `.theme-toggle` → `aria-label="Toggle theme"`, `.btn-lang` → `aria-label="Change language"`).
- Estados OK/FAIL tienen texto además del color; los screen readers lo leen naturalmente.

## 10.8 · Checklist antes de mergear

Cuando añades UI nueva, antes del PR:

- [ ] ¿Se puede tabular hasta todos los controles interactivos?
- [ ] ¿El `:focus-visible` es discernible si el control es custom?
- [ ] ¿Existe al menos un indicador no-color del estado (icono, texto, forma)?
- [ ] ¿Respetas `prefers-reduced-motion` si tu animación es ambiental?
- [ ] ¿`Esc` cierra el overlay si añadiste uno?
- [ ] ¿Los textos pequeños (`10px`, `11px`) están en grises con contraste ≥ 4.5:1?

No tenemos CI que lo verifique — es revisión manual.

## 10.9 · Lo que NO intentamos

- **Alternative stylesheets** (high-contrast mode) — si el usuario necesita más contraste, el modo oscuro ya lo provee (ratios > 12:1).
- **Traducir todos los tooltips automáticamente**: el sistema i18n existe (flask-babel, ES/EN/DE) pero los tooltips generados por JS a veces quedan en inglés. Deuda conocida.
- **ARIA live regions para cada cambio de estado**: solo para toasts importantes.
- **Focus trap modal estricto**: hoy se cierra con `Esc` y con clic fuera, suficiente en el 99% de flujos.
